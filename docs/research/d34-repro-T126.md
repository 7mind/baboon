# D34 Reproduction Record — T126

**Defect:** D34 — raw LF in MCP tool `description` string literal breaks target-language compilation
**Base commit:** `40802b4f` (T125 multi-line `describePricing` doc fixture; NO generator fix yet)
**Reproduced by:** T126 investigation (2026-06-17)

---

## Root cause

`McpDocs.flatten` returns a `String` that may contain literal `\n` characters (one per line
break in the source doc comment). Each per-backend `*McpServerGenerator` then embeds this
string inside a target-language string literal using a helper such as `csString`, `ktString`,
`javaString`, `dartString`, `swiftString`, `rustStr`. All six of those helpers escape `\\` and
`"` (or `'`) but do **NOT** escape `\n` — so the raw LF byte is copied verbatim into the
generated source file.

The `describePricing` method in `mcp-stub-ok/mcp_stub.baboon` (added in T125/D34) has a
multi-line `/** */` doc block whose `McpDocs.flatten` result spans five lines. This produces a
multi-line description string that, when embedded by the broken helpers, creates a string
literal containing raw LF characters — a compile error in six of nine backends.

**Generator helpers with the defect (missing `\n` → `\\n` substitution):**

| Backend  | Helper         | File (relative)                                     |
|----------|----------------|-----------------------------------------------------|
| C#       | `csString`     | `translator/csharp/CsMcpServerGenerator.scala:190`  |
| Kotlin   | `ktString`     | `translator/kotlin/KtMcpServerGenerator.scala:161`  |
| Java     | `javaString`   | `translator/java/JvMcpServerGenerator.scala:194`    |
| Dart     | `dartString`   | `translator/dart/DtMcpServerGenerator.scala:154`    |
| Swift    | `swiftString`  | `translator/swift/SwMcpServerGenerator.scala:184`   |
| Rust     | `rustStr`      | `translator/rust/RsMcpServerGenerator.scala:304`    |

**Generators that use JSON encoding (safe — `\n` → `\n` escape sequence):**

| Backend    | Helper           | Note                                      |
|------------|------------------|-------------------------------------------|
| Python     | `pyString`       | `Json.fromString(s).noSpaces`             |
| TypeScript | `jsString`       | `Json.fromString(s).noSpaces`             |
| Scala      | `scalaString`    | `Json.fromString(s).noSpaces`             |

## Evidence: safe backends produce properly escaped strings

The following three backends use `Json.fromString(s).noSpaces` which JSON-encodes the
description — replacing every `\n` (U+000A) with the two-byte escape sequence `\n`.

### Python — SAFE

File: `mcp/stub/McpToolsMcpServer.py`, line 32

Bytes after `Returns the fee schedule...`:
```
b'...\\nBase cost is $5 per call; \\"premium\\"...\\n\\nPass the tier name...'
```
`\\n` in Python repr = the two bytes `\` + `n` (escape sequence) = **no raw LF inside the
string literal**. The generated Python source compiles cleanly.

### TypeScript — SAFE

File: `mcp/stub/mcp-tools/mcp-server.ts`, line 19

Same encoding: `\\n` escape sequences throughout, no raw LF in the description literal.

### Scala — SAFE

File: `mcp/stub/mcptools/McpToolsMcpServer.scala`, line 52

Same encoding: `\\n` escape sequences, no raw LF in the description literal.

## Defect location summary (for T127–T130 fix tasks)

Each broken helper needs a `replace("\n", "\\n")` (and optionally `replace("\r", "\\r")`,
`replace("\t", "\\t")`) added alongside the existing `replace("\\", "\\\\")` and
`replace("\"", "\\\"")` substitutions:

- `CsMcpServerGenerator.scala:190` — `csString`
- `KtMcpServerGenerator.scala:161` — `ktString`
- `JvMcpServerGenerator.scala:194` — `javaString`
- `DtMcpServerGenerator.scala:154` — `dartString` (uses `'...'`; needs `\n` → `\n` escape)
- `SwMcpServerGenerator.scala:184` — `swiftString`
- `RsMcpServerGenerator.scala:304` — `rustStr`

The fix for Dart additionally must handle `$` → `\$` (Dart string interpolation with `$`
followed by a digit such as `$5` is not interpreted as interpolation, but `$` followed by
an identifier IS — safer to escape all `$` in single-quoted strings).

The fix for Kotlin must handle `$` → `${'$'}` or `\$` for string-template interpolation
hazard (though `$5` specifically is safe, `$tier` would be a hazard).
