## Phase 1 — Dart-from-Swift JSON failure: ROOT CAUSE FOUND

### TL;DR

The failure is **u64 overflow on Dart's `int.parse`** — NOT a missing fixture, NOT enum-related. Swift correctly emits u64 as a JSON string (`"f04":"16229105233727336919"`), Dart's decoder calls `int.parse(...)` which rejects values > 2^63-1.

### Reproducer

From `target/test-regular/dt-stub/`:
```
dart test test/testpkg/pkg0/t6_d1_test.dart
```
Output:
```
00:00 +1 ~6 -1: T6_D1 Cross-language JSON reading from swift [E]
  FormatException: Positive input exceeds the limit of integer 16229105233727336919
  dart:core                                int.parse
  lib/testpkg/pkg0/t6_d2.dart 105:44       T6_D2_JsonCodec.decode
  lib/testpkg/pkg0/t6_d1.dart 172:92       T6_D1_JsonCodec.decode.<fn>
```

### What's actually happening

1. Swift writes `u64` as a JSON **string**: `SwJsonCodecGenerator.scala:240`: `case TypeId.Builtins.u64 => q"String($ref)"`. Fixture confirms: `"f04":"16229105233727336919"`.
2. Dart decoder generated by `DtJsonCodecGenerator.scala:301`:
   ```dart
   f04: (jsonObj['f04'] is String ? int.parse(jsonObj['f04'] as String) : (jsonObj['f04'] as num).toInt()),
   ```
3. Dart `int` is signed Int64. `int.parse("16229105233727336919")` rejects positive values > 2^63-1.

### Cross-language u64 catalogue

| Lang | Encode | Decode |
|------|--------|--------|
| Scala (`ScJsonCodecGenerator.scala:345`) | `circeJson.fromBigInt(toUnsignedBigInt(ref))` | accepts both number and string |
| Kotlin | `jsonPrimitive($ref.toLong())` (signed Long; same defect class as Dart) | `$ref.toULong()` — number only |
| C# | `Value<UInt64>` (Newtonsoft writes as JSON number) | `UInt64.Parse($ref)` |
| Java | `new TextNode(Long.toUnsignedString($ref))` — JSON **string** | accepts both: `isTextual ? parseUnsignedLong : longValue()` |
| Python | JSON number (Python int unbounded) | JSON parser yields Python int |
| Rust | JSON number (default serde u64) | accepts number |
| TypeScript | `$ref.toString()` — JSON **string** | `BigInt($ref as string)` — string only |
| Dart | `$ref` — JSON number (signed Int64; loses unsignedness) | `int.parse` fails on values > Int63 max |
| Swift | `String($ref)` — JSON **string** | accepts both |

Three different conventions in production:
- **String emit, accept-both**: Java, TypeScript, Swift
- **Number emit, accept-both**: Scala, C#, Python, Rust
- **Number emit (lossy)**: Kotlin, Dart

### Recommended fix

**Option (b) — keep `int` field type, use `BigInt` as parser bridge.** Minimal disruption; semantics match Kotlin (signed view of 64-bit pattern, lossy but consistent).

- Decode: `BigInt.parse($ref as String).toSigned(64).toInt()` for string branch; `($ref as num).toInt()` retained for number branch.
- Encode: change to `$ref.toUnsigned(64).toString()` (always emit as JSON string, matching Java/TS/Swift).

### A separate path-coupling issue (out of PR-A scope)

Dart's tests reference `../../../../../target/<lang>/json-default/$typeId.json` — resolves to `<repo-parent>/target/...` (outside the repo). Works only because Swift's xcode-test-runner happens to write into a CWD that aligns. Document this; consider a `BABOON_CROSS_LANG_FIXTURE_ROOT` env var.

## Phase 2 — Cross-language enum wire-format catalogue (PR-29-D02)

### Per-backend enum encoders

| Lang | File:line | Encode form |
|------|-----------|-------------|
| Scala | `ScJsonCodecGenerator.scala:244-255` | Pascal-case via case-object `.toString` |
| Kotlin | `KtJsonCodecGenerator.scala` | enum class `.name` (Pascal) |
| C# | `CSJsonCodecGenerator.scala` | enum `.ToString()` (Pascal) |
| Java | `JvJsonCodecGenerator.scala` | enum `.name()` (Pascal) |
| Python | `PyJsonCodecGenerator.scala:97-100` | `value.value` (pydantic IntEnum/StrEnum value, currently Pascal) |
| Rust | serde `#[serde(rename_all = "PascalCase")]` or default | Pascal |
| TypeScript | `TsDefnTranslator.scala:316` | raw / configurable via `enumLowercaseValues` |
| Dart | `DtJsonCodecGenerator.scala` enum encoder | raw |
| Swift | `SwJsonCodecGenerator.scala` | raw |

### Decoder case-sensitivity

- C#: `Enum.TryParse(asStr, true, ...)` — case-insensitive
- Java: case-sensitive
- Scala: case-sensitive (companion `parse(...)`)
- Kotlin: case-sensitive (`enumValueOf`)
- Python: exact match
- TS / Dart / Swift: exact match

### Recommendation: emit Pascal-case source-name as canonical wire format

- 6/9 backends already emit Pascal naturally; forcing raw needs `toString` overrides + serde-rename overrides
- 3 backends (TS/Dart/Swift) emit raw and would change to Pascal — minor codegen-line changes
- All current tests use Pascal-case enum source names — change is identity for the dominant test suite
- Avoids the Scala case-objects-must-capitalize ergonomic issue

### Spec

1. Wire form: `m.name.capitalize` (first letter upper, rest unchanged) by default.
2. Optional CLI flag `--<lang>-enum-wire-style=pascal|lowercase|raw` (default `pascal`).
3. All decoders must accept exact match (no case-insensitive permissiveness — make it tight). C# drops the `true` arg in `TryParse`.
4. Single shared helper in a common place (e.g. `BaboonEnquiries` or new `EnumWireStyle.scala`) — all 9 backends route through it.

### PR-27-D01 — Scala CopyEnumByName conversion

- `ScConversionTranslator.scala:188-191`: `mappingEntries` keyed by raw source-strings
- Then at line 197: `Map(...).getOrElse(from.toString, from.toString)` — but `from.toString` of Scala case-object yields capitalized
- **Fix**: capitalize the keys to match `from.toString`, OR build the mapping using the same transform as case-object emission. Cleanest: keys + the `from.toString` lookup expression both go through the same wire-form helper from PR-E.
- Dormant because `T_NsPascal` was removed from the fixture in M14.

### PR-28-D04 — Kotlin self-cast `(X as X).decodeBranch(...)`

- `KtUEBACodecGenerator.scala:196`: `q"""($cName as $cName).decodeBranch(ctx, wire)"""`
- Only emitted in `wrappedAdtBranchCodecs=true` mode (test-kotlin-wrapped exercises this)
- USELESS_CAST doesn't fire — likely Kotlin compiler quirk on companion-object self-cast
- **Fix**: drop parens and `as $cName`. Replace line 196 with `q"""$cName.decodeBranch(ctx, wire)"""`.

## Cross-cutting risks

- **Regenerated test fixtures:** PR-32's Dart u64 change alters JSON wire output of every Dart-emitted fixture in `target/{test-regular,test-wrapped}/target/dart/json-default/*.json`. Other backends' cross-language tests already see Dart-output as input. Mitigation: stage so encode-side update doesn't break decoders in other backends.
- **conv-test/* impact:** baselines compile against the previous wire form. Run `:test-gen-compat-<lang>` for each language after PR-32 and PR-35.
- **CLI flag growth:** PR-35 adds 9× language flags or one global. Document in `CLIOptions.scala` with `@HelpMessage`.
- **Test cwd-coupling (Phase 1 finding):** Dart's `../../../../../target/<lang>/json-default/...` paths happen to work only because of a fragile CWD-coupling. Document; not blocking.
