## M18 — BAB-A01 identifier types

### Q-M18-1 ⚑ — Reverse-parse scope
Should `parse(str): Self` (or `tryParse`/`fromString`/`Self::from_str`) be in-scope for M18?

- (a) All 9 languages — biggest effort, ~doubles M18 size.
- (b) Scala only as reference implementation — recommended; validates the escape scheme, gives users one canonical impl to consult.
- (c) Deferred entirely to a later milestone.

**Recommendation:** (b).
**Answer:** all 9 languages, we need that for map keys (see below)

### Q-M18-2 — All-9-language toString rollout
- (a) Bundle into one PR — recommended; toString is mechanical, ~50 LoC per backend.
- (b) Stage one PR per language — like M2..M10 codecs.

**Recommendation:** (a) bundled.
**Answer:** (a)

### Q-M18-3 — `id` placement scope in M18
- `root id` — allowed.
- `id` inside `ns` — allowed.
- `id` inside `adt` branches — **rejected in M18, deferred to M20** (would force `Typedef.Adt.dataMembers` filtering changes; cleaner to defer).

**Recommendation:** above.
**Answer:** defer to M20

### Q-M18-4 ⚑ — Pkg/owner in toString repr
Format is `<Name>:<ver>#field:value:...`. The `<Name>` segment can be:

- (a) Simple type name (`UserId`) — recommended; doesn't leak internal namespacing.
- (b) Full FQN (`pkg.owner.UserId`) — bloats repr but disambiguates across packages.

**Recommendation:** (a) simple name.
**Answer:** (a) simple name

### Q-M18-5 ⚑ — Versioned vs unversioned repr
Should `id` types ship:

- (a) One `toString` (versioned canonical) — recommended for M18.
- (b) Two methods: `toString` (versioned) + `toUnversionedString` (for storage keys, URL components, log lines).
- (c) Annotation-driven opt-in: `: derived[id-stable]` or similar to choose unversioned.

**Recommendation:** (a) for M18; revisit if real demand emerges.
**Answer:** (a), no need to revisit, this form is not inteded to be used in versioning workflows and while we want it to be parseable we do NOT expose parsers now in an easy to use form

### Q-M18-6 — Float scope clarification
Spec listed "primitive types" without enumerating. Proposal:

- **Allowed**: `bit`, `i08`, `i16`, `i32`, `i64`, `u08`, `u16`, `u32`, `u64`, `str`, `uid`, `tsu`, `tso`, `bytes`.
- **Rejected**: `f32`, `f64`, `f128` (locale-sensitive toString, not bit-stable cross-language).

Specifically: `tsu`/`tso` (timestamps) **allowed** — they have unambiguous string form (RFC 3339).

**Recommendation:** above. `bytes` rendered as lowercase hex (no separators).
**Answer:** yes, above

### Q-M18-7 — GraphQL/OpenAPI emission
Identifiers emit as:

- (a) Object types (same as `data`) — recommended for M18, no special-casing.
- (b) Scalar strings using the toString repr — richer integration, but requires GraphQL custom scalar / OpenAPI string-with-pattern.

**Recommendation:** (a) for M18; (b) as opt-in feature in a later milestone.
**Answer:** (a)

### Q-M18-8 — Always-derive JSON+UEBA?
- (a) Implicit always-derive — typer auto-attaches `derived[json]` and `derived[ueba]` to every `id` type. No syntax overhead.
- (b) Explicit — user writes `id UserId : derived[json,ueba] { ... }`.

**Recommendation:** (a). Identifiers as a primitive-extension category should be wire-ready by default.
**Answer:** derivation rules should be the same as for data classes - so depends on annotations and compiler flags

### Q-M18-9 — Reserved-word check
`id` is short and might conflict with user field names. Field names live in different syntactic position (after `:` in field defs), so no parser conflict. Other reserved-word concerns?

**Recommendation:** none anticipated; the `kw` parser uses `!identChar` boundary so `id` only matches the bare token (won't collide with `idx`, `id_foo`, etc.).
**Answer:** our keywords are not global, they are context-dependent, "id" is fine - it still should allow "id" as field name

## M19 — BAB-A02 user-DTO map keys

### Q-M19-1 — Wrapper-of-wrapper recursion depth
`data StrongUserId { v: UserId }; data UserId { v: uid }` — recurse twice through wrapper chain to reach `uid`.

**Recommendation:** yes, no depth limit beyond cycle detection.
**Answer:** yes, no depth limit, detect cycles (existing loop detection/dependency mechanics which is based on adjacency lists should work)

### Q-M19-2 — Floats as wrapper-key inner type
`data Score { v: f64 }` as map key. Current builtin path already accepts `f32`/`f64`/`f128` keys.

- (a) Keep parity — recommended; accept float wrappers iff floats accepted as builtin keys.
- (b) Tighten globally — reject all float map keys (builtin AND wrapper), call out as breaking change in migration notes.

**Recommendation:** (a) parity.
**Answer:** reject floats for wrappers, allow builtin

### Q-M19-3 — Both shapes coexist
Wrapper's standard JSON-object codec (value-position, e.g. `{"value": "abc"}`) PLUS new map-key codec (key-position, e.g. `"abc"` directly). Both shapes coexist; emitted in parallel by codegen.

**Recommendation:** confirm — both needed.
**Answer:** both needed obviously! wrapper might be used in different positions!

### Q-M19-4 ⚑ — Eligibility: root-data only?
Your spec said "root-data wrappers". Non-root data can transitively appear in maps today (via `lst[Foo]`).

- (a) Scope eligibility to **root-data only** — recommended; matches spec exactly, simpler rule.
- (b) Allow any single-primitive-field `data` — broader.

**Recommendation:** (a) root only.
**Answer:** (b)

### Q-M19-5 — `opt`-around-primitive in wrapper field
`data MaybeId { v: opt[uid] }` — option-around-primitive has no stable key form for the absent case.

**Recommendation:** reject — ineligible as map key.
**Answer:** reject

### Q-M19-6 — M18 sequencing interaction
With M19 shipping first: `id` types automatically inherit map-key eligibility for single-primitive `id` shapes. Multi-field `id` types remain ineligible.

**Recommendation:** ship M19 before M18.
**Answer:** multi-field ids should remain eligible - if we have parsers from string representation, that works just fine

### Q-M19-7 — Foreign types with `runtimeMapping`
Foreign types with `runtimeMapping = Some(target)` recurse into the runtime-mapped target both in codegen and runtime codec.

**Recommendation:** same rule applies — a wrapper around a Foreign whose `runtimeMapping` resolves to a primitive is eligible.
**Answer:** we don't know what foreign types resolve into (we don't know anything about them in baboon), so as with regular foreigns, we could accept any single-foreign data class but that's user's responsibility

### Q-M19-8 — Validator vs codegen failure mode
Today: model rejected at codegen with `BUG: Unexpected key usertype`. After M19: ineligible wrapper rejected at validator time with clean error message. The codegen `BUG:` throws are kept as defensive assertions (validator should have caught it).

**Recommendation:** above.
**Answer:** agreed

## M20 — BAB-A03 ADT branch inheritance

### Q-M20-1 ⚑ — Inclusion semantics: re-emit vs reference
**The biggest M20 decision.**

- (a) **Re-emit** (recommended): `+ ErrorAtom` desugars to structural copies of each branch, owned by the receiving ADT. `SomeError.Forbidden` and `ErrorAtom.Forbidden` are **distinct `TypeId.User`**. Wire format / codegen / schema emission untouched. Cross-ADT conversion is explicit. Pure source-level sugar.
- (b) **Reference**: branches become first-class types shared across ADTs. `SomeError` and `ErrorAtom` literally share the same `Forbidden` type. Cross-ADT conversion is identity. **But:** breaks the existing `Owner.Adt(adtId)` invariant, requires a new owner kind, makes wire-format discriminators ambiguous when a branch is shared by N ADTs, and changes evolution semantics. Significantly bigger change with wire-format implications.

**Recommendation:** (a) re-emit. (b) is its own milestone if ever pursued.
**Answer:** (a) - we should just insert copies of each branch at AST level - that's the easies approach

### Q-M20-2 ⚑ — Name-collision policy
`SomeError` declares its own `Forbidden` AND `+ ErrorAtom` (which has its own `Forbidden`). Result?

- (a) **Error** (recommended): reject with `DuplicatedAdtBranches`. User can opt in via explicit `- ErrorAtom.Forbidden` (or `- Forbidden` shorthand) before the include.
- (b) Last-wins silently.
- (c) Must-be-identical-shape (allow only if structurally equal).

**Recommendation:** (a) error.
**Answer:** reject, we fail on collision. No dot-prefixed syntax (easiest)

### Q-M20-3 — Selective inclusion `+ X { Foo, Bar }`
Pull only specific branches by name?

- (a) Not in M20 — recommended; field-side composition doesn't support name-selection. Use `- X.Foo` shorthand to subtract instead.
- (b) Add `+ X { Foo, Bar }` syntax — extra grammar, asymmetric with field-side.

**Recommendation:** (a) not in M20.
**Answer:** (a) - we don't need this feature at all. Agreed on subtraction

### Q-M20-4 — Empty intersection
`adt X { ^ A; ^ B }` where `A` and `B` share no branch names → empty branch set.

- (a) Treat as `EmptyAdt` error — recommended; consistent with current empty-ADT rejection.
- (b) Allow empty.

**Recommendation:** (a) error.
**Answer:** error

### Q-M20-5 — Chained inclusions
`adt A { + B }`, `adt B { + C }`. Should `A` transitively get `C`'s branches?

**Recommendation:** yes, automatically. With re-emit + toposort-ordered processing, by the time `A` is processed, `B` already has `C`'s branches absorbed.
**Answer:** accept, we just need to expand our dependency extractor and make an early pass adding the branch copies

### Q-M20-6 — Annotation copying on re-emit
When a branch is re-emitted into the receiving ADT, do per-branch `derived[json]`/`derived[ueba]` annotations propagate?

**Recommendation:** yes, mirror source.
**Answer:** yes

### Q-M20-7 — Position of include lines
Where can `+ X` / `- X` / `^ X` lines appear inside an `adt` body?

- (a) **Anywhere** among branch declarations — recommended; mirrors DTO field-side rules.
- (b) Only at the top of the body, before any branch.

**Recommendation:** (a) anywhere.
**Answer:** anywhere

### Q-M20-8 — `is X` (contract) + `+ Y` (ADT include) interaction
Contracts impose required fields on every branch; includes contribute additional branches that must satisfy those fields. New branches in the receiving ADT must additionally satisfy *its* contracts (caught by existing `MissingContractFields` path).

**Recommendation:** above interpretation.
**Answer:**  yes, your interpretation. We should get that automatically if we insert branches at AST level in an early stage of the typer

## Follow-up round (2026-04-29 0950+)

### Q-FU-1 — `id` / wrapper as map key without JSON derivation
**Answer:** (a) Validator rejects. `id`/wrapper types used as JSON map keys must have `derived[json]` (and `derived[ueba]` if used in UEBA). Same rule applies to plain `data` wrappers. Parsing/repr machinery is intrinsic to the type identity but JSON map-key codec emission counts as a derived feature.

### Q-FU-2 — M20 subtraction syntax
**Answer:** `- X.Foo` (dot-prefixed single-branch removal) accepted. `- X` (whole-ADT removal) accepted. `+ X.Foo` (selective inclusion) is **optional** — implement only if easy, otherwise skip; not needed feature.

### Q-FU-3 — M20 expansion stage
**Answer:** (b) typer-early-pass acceptable. Suggested location: in `runTyper`, once types are ordered by toposort but before the main `foldLeft` — re-evaluate during execution if a different positioning makes the implementation easier.

### Q-FU-4 — M18 parser visibility
**Answer:** (b) parsers live on a separate codec/internal namespace, not as primary public API on the type. Goal: discourage users from using the toString form unless necessary; map-key codec emission can still reach the parser.
