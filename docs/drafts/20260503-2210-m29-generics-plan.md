## 1. Goal and scope

BAB-A04 delivers **user-defined generic templates** to the `.baboon` DML:
declarations of the form `data X[T] { … }`, `adt X[T] = …`, `contract X[T] = …`,
`service X[T] { … }` (henceforth "templates") together with **alias-only
instantiation** `type Y = X[Foo]` (henceforth "instantiation"). The compiler
**monomorphises** every reachable instantiation into an ordinary `Typedef.User`
keyed by the alias's `TypeId.User`, leaving every downstream stage — validator,
codec generation, evolution diff, conversion emit, all 9 codegen backends, JSON
and UEBA wire formats — operating on the same `DomainMember` shapes they handle
today. Templates themselves never become `DomainMember`s; they live in a typer-
side registry consumed by the monomorphisation rewrite and discarded before
codegen.

**Not in scope this milestone** (full list in §7): higher-kinded templates,
variance annotations, where-clauses / type-class bounds, defaulted type-params,
field-position instantiation, nested instantiation in alias RHS (locked decision
#3), template recursion/self-reference (locked decision #2), per-language reified
generics, evolution of templates as templates (only the materialised instances
evolve, locked decision #5).

## 2. Locked decisions (input — verbatim from tasks.md)

The following six numbered decisions and the nine-case negative-test matrix are
user-blessed (2026-05-03) and not to be re-litigated. Any deviation discovered
during execution must be raised to the user before implementation.

### 2.1 Six locked design decisions

> 1. ADTs, Contracts, Services are all generifiable. AST-level templating handles
>    them uniformly.
> 2. Self-reference is forbidden. Only DAG-shaped template-and-instantiation
>    graphs are valid.
> 3. The **only** way to instantiate is through a type alias. Field-position
>    instantiation and nested instantiation in alias RHS are forbidden —
>    intermediate aliases required.
> 4. Monomorphisation identifier = alias id. No synthetic id `X<i32>` exposed.
> 5. Templates not subject to migration; instances (concrete `Typedef.User`
>    under alias id) are.
> 6. `: derived[…]` written **only on the alias**, propagates to monomorphisation.

### 2.2 Negative-test matrix (locked)

> 1. `field: Foo[Bar]` — instantiation in field position.
> 2. `type Y = Foo[Bar[i32]]` — nested instantiation in alias RHS.
> 3. `data X[T] { a: lst[T] }; type XInt = X[i32, str]` — arity mismatch.
> 4. `data X[T, T] { … }` — duplicate type-param name.
> 5. `data X[T] { rec: X[T] }` — self-reference (forbidden).
> 6. `field: T` outside a template — placeholder leak.
> 7. `type Y = X` — template referenced without instantiation.
> 8. `type Y = i32[X]` — instantiating a non-generic.
> 9. `type A = X[B]; type B = X[A]` — cycle through aliases.

### 2.3 Cross-cutting architectural notes carried in from M28 (locked)

Wire-format invariants from M28-N01/N02 remain in force; M29 must not perturb
them. Specifically: the new `m29-ok` acceptance fixture must round-trip JSON and
UEBA across all 9 backends without producing a single fixture-md5 churn outside
explicitly added/changed fixtures (§5.2).

## 3. Architectural shape (verified)

This section describes the end-to-end pipeline once templates exist. Every claim
below has been verified against source by direct read; the file path and the
verified line range follow each claim.

### 3.1 Where templates enter the AST (parser)

- `BaboonParser` already parses `Foo[Bar]` syntax in field position via
  `RawTypeRef.Constructor(name, params, prefix)` —
  `baboon-compiler/src/main/scala/io/septimalmind/baboon/parser/defns/DefDto.scala`
  L53–79 (`typeRef`). The `[T]` head on a *declaration*
  (`data X[T] { … }`) does **not** parse today: `DefDto.dtoEnclosed` (L148–153),
  `DefAdt.adtEnclosed` (L57–66), `DefContract.contractEnclosed`,
  `DefService.service` (L17–22) all read a bare name via `meta.member`, then
  parse the body. M29 adds an optional `[T1, T2, …]` clause between the name
  and the body for these four declaration kinds.
- The right-hand side of an alias is `RawTypeRef`
  (`RawAlias(name, target: RawTypeRef, meta)` —
  `baboon-compiler/src/main/scala/io/septimalmind/baboon/parser/model/RawDto.scala`
  L32). `RawTypeRef.Constructor` already covers the parse shape `X[Foo]`. No
  parser change is needed for the alias RHS itself; M29 adds a typer-side rule
  that `RawTLDef.Alias.value.target` may be `RawTypeRef.Constructor` only when
  it instantiates a template (today this construction parses but always fails
  in the typer because `Constructor.id` resolves only to `BuiltinCollection`).

### 3.2 Raw AST representation of a template head (proposed)

Two options. The plan picks (b) and the orchestrator should challenge before
execution if they prefer (a):

- **(a)** Add a parallel sealed branch under `RawDefn` per template kind:
  `RawTemplateDto`, `RawTemplateAdt`, `RawTemplateContract`,
  `RawTemplateService`. Pros: type-system enforces that template-only
  positions never see a non-template node. Cons: every existing pattern match
  on `RawDefn` (and there are many — `ScopeBuilder`, `BaboonTyper.runTyper`,
  `AdtInheritanceExpander`, `TyperIssue.ScopeCannotBeEmpty.memberType`) gains
  four new cases.
- **(b) — proposed** — Extend `RawDto` / `RawAdt` / `RawContract` / `RawService`
  with an optional `typeParams: List[RawTypeName]` (default `Nil`).
  `typeParams.isEmpty` is the existing non-template case; `typeParams.nonEmpty`
  marks a template. Pros: minimum churn — every site that processes the
  non-template case continues to read the same field shapes; only the typer-
  early registry pass and the validator's "template-only" filter need to
  branch. Cons: structurally legal nonsense (`typeParams.nonEmpty` on a node
  that the validator does not yet recognise) is only caught at the next pass.

  Mitigation: the typer-early "register templates" pass (PR-29.4) runs **before**
  `BaboonTranslator.convertX` is asked to produce `Typedef.User` for a node, so
  templates are extracted out of the `members` list before `convertDto` etc. ever
  see them. A non-template path that accidentally meets `typeParams.nonEmpty` is
  a typer bug; raise a `BaboonBug`-flavoured `TyperIssue`.

  Selection rationale: (b) keeps PR-29.2 narrow to a parser change + one new
  field on four case classes, and isolates the structural divergence to the new
  template-registry pass. The exhaustive-match audit footprint is smaller.

### 3.3 Template registry (typer)

A new typer-early pass `TemplateRegistryBuilder` scans the rewritten raw AST
**before** scope-building begins (i.e. *before* the existing
`adtInheritanceExpander.expand` call at
`baboon-compiler/src/main/scala/io/septimalmind/baboon/typer/BaboonTyper.scala`
L406). For each `RawDto`/`RawAdt`/`RawContract`/`RawService` with
`typeParams.nonEmpty` it:

1. Validates the type-param list (non-empty, no duplicates → matrix #4).
2. Stores the raw definition body keyed by `(pkg, owner, name)` in a per-domain
   `TemplateRegistry`.
3. **Removes** the template node from the list of `RawTLDef`s that downstream
   passes see, so that no `DomainMember` is ever produced for the template
   itself (locked decision #4).

The template body is kept as a raw AST (post-parse, pre-typer-resolve). Type
parameters appear in field-position `RawTypeRef.Simple(RawTypeName("T"), Nil)`
nodes; substitution happens at instantiation time as a structural rewrite over
the cached body.

### 3.4 Instantiation and monomorphisation (typer)

A second new pass `TemplateInstantiator` runs **after** the template registry is
built but **before** the second `builder.buildScopes` invocation (post-ADT-
expansion at `BaboonTyper.scala` L413). It:

1. Walks every `RawTLDef.Alias` in the (post-ADT-expansion) member list.
2. For each alias whose `target` is `RawTypeRef.Constructor(name, args, prefix)`
   pointing at a registered template:
   - validates arity (`args.size == template.typeParams.size` → matrix #3);
   - validates that no `args(i)` is itself a `RawTypeRef.Constructor` over a
     template (nested instantiation forbidden, matrix #2; nested over a builtin
     collection like `lst[T]` remains legal);
   - performs **AST substitution** over the cached template body, replacing
     every `RawTypeRef.Simple(RawTypeName(p), Nil)` whose `p` matches a type
     parameter with the resolved argument `RawTypeRef`;
   - synthesizes a fresh `RawDto` / `RawAdt` / `RawContract` / `RawService`
     whose `name` is the **alias's** name and whose `meta` is the alias's meta
     (locked decision #4: monomorphisation id = alias id);
   - propagates `derived` from the alias node to the synthesized definition
     (locked decision #6);
   - **replaces** the `RawTLDef.Alias` in the member list with the corresponding
     concrete `RawTLDef.{DTO|ADT|Contract|Service}` carrying the synthesized
     definition.
3. Aliases whose `target` is `RawTypeRef.Simple(name, prefix)` and whose
   resolved scope target is **also** a template (i.e. naked reference without
   `[…]`) emit `TemplateNotInstantiated` (matrix #7).
4. Aliases whose `target` is a non-template `RawTypeRef.Constructor` (e.g.
   `type Y = i32[X]`) emit `NotATemplate` (matrix #8).
5. `RawTypeRef.Constructor` subtrees in **any** non-alias position whose head
   resolves to a template emit `TemplateInstantiationOutsideAlias` (matrix #1
   for fields; same diagnostic catches matrix #2 via the nested check).
6. After all aliases are substituted, the rewritten member list contains only
   ordinary (non-template, non-template-aliased) declarations. The existing
   `BaboonTyper.runTyper` loop from L408 onwards runs unmodified.

Cycle detection (matrix #9 — `type A = X[B]; type B = X[A]`) is handled by the
existing toposort over the rewritten member list in
`BaboonTyper.order` (L445–468), because after substitution `A` and `B` become
ordinary `RawDto`s whose fields reference each other → the `BadInheritance` /
`CircularInheritance` machinery already covers it. **Hypothesis to verify in
PR-29.5:** the existing cycle path produces a usable error message at this
location; if the message references "inheritance" rather than "alias instantiation"
the wording will need to widen, or a new diagnostic added pre-toposort.

### 3.5 Why codegen is unchanged (verified implication of decision #4)

Verified: all 9 backends operate on `Domain.defs` keyed by `TypeId`, walking
`TypeRef.Constructor(id: TypeId.BuiltinCollection, args)` recursively. After
monomorphisation, every reachable type is an ordinary `DomainMember.User` keyed
by the alias's `TypeId.User`; the only `TypeRef.Constructor` nodes in any
field's `tpe` are over `BuiltinCollection` (lst/map/opt/set), exactly as today.
Therefore PR-29.5 requires zero changes under `translator/{csharp,scl,python,
rust,typescript,kotlin,java,dart,swift,graphql,openapi}/`.

This claim is the central architectural bet of the milestone. It is verifiable
mechanically: the cross-language emission smoke fixtures (PR-29.9) and the
acceptance fixture (PR-29.10) pass without any backend change touched in the
diffs.

### 3.6 The TypeRef new-variant-vs-widen question (architectural decision)

The recon flagged this as the single most consequential architectural choice.
Two options:

- **(W)** Widen `TypeRef.Constructor.id` from `TypeId.BuiltinCollection` to a
  new shared supertype that admits `TypeId.User`. Add a `TypeRef.Constructor`
  whose `id` is a user template-id, carried through to codegen. Requires every
  backend to learn how to render parameterised user types, which violates
  decision #4 (no synthetic id exposed) and explodes the milestone scope.
- **(N)** Add a new `TypeRef` variant `TypeRef.UserConstructor(id: TypeId.User,
  args: NEList[TypeRef])` that exists **only as an intermediate** carried by
  the typer-side substitution machinery, never reaching codegen. Requires every
  pattern match on `TypeRef` to gain a case.
- **(K) — proposed** — *Keep `TypeRef` exactly as it is today.* Templating
  happens at the **raw** layer (`RawTypeRef.Constructor` already supports
  `X[Foo]`). Template parameters live as `RawTypeRef.Simple(RawTypeName("T"),
  Nil)` placeholders inside cached template bodies; substitution rewrites the
  raw AST in `TemplateInstantiator`; the resulting raw AST contains only
  non-template `RawTypeRef` shapes by the time `BaboonTranslator.convertTpe`
  (`baboon-compiler/src/main/scala/io/septimalmind/baboon/typer/BaboonTranslator.scala`
  L437–505) sees it.

  Selection rationale: option (K) is the strict consequence of locked decisions
  #3 (alias-only instantiation) and #4 (alias id is the canonical id). Because
  instantiation never appears in any non-alias position in valid input, and the
  alias rewrite happens *before* `convertTpe` is asked to type-check the alias
  body, no `TypeRef`-level representation of "user-defined template
  instantiation" is ever needed. The post-substitution raw AST is structurally
  indistinguishable from a hand-written non-template equivalent.

  This is also what makes claim §3.5 true: if `TypeRef` never gains a new
  variant, no backend pattern-match needs to learn anything.

  Risk: any path that escapes the substitution rewrite (a template-parameter
  `T` reaching `convertTpe`) would produce a `NameNotFound` from
  `scopeSupport.resolveTypeId` rather than a precise diagnostic. PR-29.7 adds a
  pre-substitution validator that catches `T` in non-template position (matrix
  #6) so the `convertTpe` path never sees an un-substituted placeholder.

**Decision recorded:** option (K) — no `TypeRef` change.

### 3.7 Pipeline placement diagram

```
parse → RawTLDef tree
  ↓
TemplateRegistryBuilder (NEW, PR-29.4)
  · extracts templates into registry
  · returns member list with templates removed
  ↓
[existing] AdtInheritanceExpander.expand   ← already at BaboonTyper.scala L406
  ↓
TemplateInstantiator (NEW, PR-29.5)
  · rewrites RawTLDef.Alias whose RHS is a template instantiation
  · validates arity, nesting, non-template instantiation, etc.
  ↓
[existing] builder.buildScopes (second invocation, L413)
[existing] order / toposort (L416)            ← catches matrix #9
[existing] convertX → DomainMember.User       ← unchanged
[existing] resolveAliasInfo for non-template aliases
  ↓
Domain → validator → codegen   ← all unchanged
```

This is structurally identical to the slot M20's `AdtInheritanceExpander` already
occupies. The new passes follow the same `Error2` / `BaboonIssue` accumulation
pattern (`F.traverseAccumErrors`).

  - `baboon-compiler/src/main/scala/io/septimalmind/baboon/typer/TemplateInstantiator.scala`
    (new) — pure rewriting pass over `Seq[RawTLDef]` with the
    `TemplateRegistry` from PR-29.4 in scope. Substitution algorithm:
    1. Find every `RawTLDef.Alias(_, RawAlias(name, target, meta))`.
    2. If `target` is `RawTypeRef.Constructor(tname, args, prefix)` and
       `(pkg, owner-of-this-alias, tname)` (or a scope-resolved equivalent
       allowing `prefix`) hits the registry, substitute.
    3. Substitution: walk the cached raw body. Every
       `RawTypeRef.Simple(RawTypeName(p), Nil)` whose `p` is a type-param name
       in the template's `typeParams` is replaced by the corresponding
       resolved `RawTypeRef` from `args`. Walk the body recursively
       (`RawTypeRef.Constructor.params` → recurse;
       `RawTypeRef.AnyRef.underlying` → recurse). Field metadata and
       `derived` annotations on the body are preserved verbatim.
    4. Pick the synthesized definition's name from the alias's `name`. Pick
       its `meta` from the alias's `meta`. Pick its `derived` from the
       alias's `RawTLDef.Alias` meta (decision #6).
    5. Replace the `RawTLDef.Alias` with the corresponding
       `RawTLDef.{DTO|ADT|Contract|Service}` carrying the synthesized def.
    6. Aliases NOT pointing at a template pass through unchanged.

## 5. Cross-cutting risks and mitigations

1. **TyperIssue 3-site exhaustive match.** Every PR adding a `TyperIssue`
   case (PR-29.4, PR-29.6, PR-29.7) MUST update all three exhaustive
   sites (`DiagnosticsProvider.extractTyperIssueInfo`,
   `WorkspaceState.extractTyperIssuePointer`,
   `BaboonJS.extractTyperIssuePointer`). The `WorkspaceState.formatIssue`
   site has a `case _ =>` fallback and does NOT need updating, but adding
   a precise case there improves user-facing error messages in the LSP.
   Mitigation: bundle multiple new cases in a single touch per file
   (PR-29.7 specifically) and verify with `mdl --seq :build :test` (the
   `+compile` cross-build catches JS-side misses).

2. **Fixture md5 churn.** Wire-format invariants from M28-N01/N02 require
   that **existing** fixtures (`pkg0`, `m19-ok`, `m20-ok`, `m28-ok`,
   `any-ok`, `identifier-ok`, `m20-was-propagation`, `rename-ns`)
   produce byte-identical output through M29. Mitigation: each PR's
   close gate explicitly checks `git diff --stat target/test-regular/
   target/test-wrapped/` and verifies only `m29-*/` paths appear in the
   diff.

3. **ADT inheritance + generics interaction.** `DefAdt` currently uses
   `nonGenericTypeRef` for `+ X` / `- X` / `^ X` arms (verified
   `parser/defns/DefAdt.scala` L28, L36, L44). M29 does NOT widen this
   to allow `+ X[T]` arms — the spec PR-29.1 must say so explicitly.
   The interaction with templates is: **a template ADT** (`adt X[T] = …`)
   can use ADT-inheritance arms in its own body, but the arms must
   themselves be non-template references. Substitution then materialises
   the arms with the substituted type-param. Verify in PR-29.5 with a
   fixture covering this case.

4. **Monomorphisation must run before codegen — placement in the typer
   pipeline.** Verified placement: between `AdtInheritanceExpander.expand`
   (`BaboonTyper.scala` L406) and the second `builder.buildScopes` call
   (L413). PR-29.4 inserts `TemplateRegistryBuilder` *before* L406;
   PR-29.5 inserts `TemplateInstantiator` *between* L406 and L413. This
   ordering ensures: (a) ADT inheritance arms can reference templates'
   monomorphised forms (because templates are substituted *after* ADT
   expansion completes); (b) the second `buildScopes` invocation sees
   only ordinary `RawDto` / `RawAdt` / etc. nodes, so all downstream
   passes — including `convertTpe`, validator, codegen — operate on
   the same shapes they handle today.

5. **Self-reference vs. template-recursive references.** Locked decision
   #2 forbids self-reference. Matrix #5 (`data X[T] { rec: X[T] }`) is
   the canonical case. Open question (raise to user before PR-29.5
   lands): does a *transitive* shape like `data Tree[T] { children: lst[Tree[T]] }`
   also count as forbidden? If yes (likely intended reading of
   "DAG-shaped"), PR-29.7's `TemplateSelfReference` detection must walk
   the body recursively. If no (allowing recursive containers), the
   detector limits to direct-self only and recursive container shapes
   are valid. The plan defaults to YES (forbid both) per the strict read
   of decision #2; raise as the only meaningful open question (§6).

6. **Cross-build `sbt +compile` discipline.** CLAUDE.md historical
   failure: PR-47 (M21) shipped green local `sbt baboonJVM/compile` but
   broke CI on `BaboonJS.scala`. M29 PRs must run `mdl --seq :build :test`
   not `sbt baboonJVM/compile`. The `mdl :build` target triggers
   `sbt +compile` which is the same path CI uses.

7. **Resource-file caching.** Generated runtime resources under
   `baboon-compiler/src/main/resources/baboon-runtime/` are cached by
   the `PortableResource.embedSources` macro. M29 should not need to
   touch any runtime file (decision §3.5 — codegen unchanged), but if
   it does (say, a new helper function for a backend), the executor
   MUST `sbt clean` before `sbt compile` per CLAUDE.md.

8. **Local Kotlin OOM workaround.** `mdl :test` runs language test
   actions in parallel by default; on <16 GB RAM laptops the Kotlin
   compiler daemon OOMs. Use `mdl --seq :build :test` for executor
   close gates (default sequential). CI uses default parallelism.

9. **Service-flavour RPC wire compatibility.** Service templates produce
   monomorphised services with method signatures referencing the
   substituted types. PR-29.10's `:test-service-acceptance` MUST exercise
   at least one templated service to prove the RPC wiring round-trips.

10. **AnyOpaque interaction (`any[T]`).** The `any[T]` builtin (M28
    family) accepts any type as its underlying. After monomorphisation,
    `any[X[i32]]` would have to be encoded as `any[IntX]` (the alias
    id). Verify that the `RawTypeRef.AnyRef.underlying` field is walked
    by PR-29.5's substitution pass; if a template-parameter `T` appears
    inside an `any[T]`, substitution must descend into it. Add a
    fixture in PR-29.9 that exercises `any[T]` inside a template.

11. **`derived[…]` propagation timing.** Decision #6: `derived` written
    only on the alias propagates to the materialised definition.
    PR-29.5 must NOT preserve any `derived` annotations on the template
    body itself (template-side `derived` is a spec error per decision #6
    — emit `TemplateCarriesDerivation` as a defensive diagnostic, or
    silently strip and document). Resolve in PR-29.1's spec.

## 6. Open questions for the user

Only one question survives the locked decisions and recon:

1. **Recursive-container template bodies.** Locked decision #2 says
   "Self-reference is forbidden. Only DAG-shaped template-and-instantiation
   graphs are valid." The matrix #5 case is direct
   self-reference (`data X[T] { rec: X[T] }`). Does the same prohibition
   extend to **container-mediated** self-reference like
   `data Tree[T] { children: lst[Tree[T]] }`?
   - **Option A (strict):** forbid all template-self-reference,
     direct or container-mediated. PR-29.7's `TemplateSelfReference`
     walks the body recursively. Materialised `Tree`-of-leaves needs
     a hand-written non-template recursive type if needed.
   - **Option B (permissive):** allow container-mediated self-reference
     (covered by the existing `terminatesLoop` check in
     `BaboonValidator.checkLoops`, L89, which already accepts
     container-mediated cycles for non-template recursive types like
     a hand-written `Tree`). PR-29.7 limits `TemplateSelfReference` to
     direct field-position recursion only.

   The plan defaults to **Option A** (strict) per the literal reading of
   decision #2. If the user intends Option B, PR-29.7's diagnostic
   narrows and the spec PR-29.1 must say so explicitly.

   No other open questions identified — the locked decisions plus the
   verified recon answer everything else (template-on-identifier handling
   resolved as "out of scope this milestone" per §7; alias-on-alias
   verified to pass through `scopeSupport.resolveAlias` unchanged in §3.7
   and PR-29.5 risk callout).

## 7. Out of scope (explicit)

The following are deliberately **NOT** delivered in M29 and will require
separate milestones/decisions:

1. **Templated identifiers** (`id Foo[T] { … }`). The spec narrative for
   identifiers (PR-55, M18) restricts identifier fields to a tightly
   controlled type set; templating opens questions about the
   substituted-id's wire form (M28-N01 / map-key encoding) that are
   independent of M29's monomorphisation question. Default behaviour:
   PR-29.2 parses `id` declarations *without* `[…]` only; a parse
   failure rejects `id Foo[T] { … }`.

2. **Higher-kinded templates** (`X[F[_]]`). The substitution algorithm in
   PR-29.5 requires every type-param to be substituted by a concrete
   `RawTypeRef`. Higher-kinded substitution is a separate type-theoretic
   feature that we do not need for the alias-only-instantiation surface.

3. **Variance annotations** (`X[+T]`, `X[-T]`). The monomorphisation
   model produces concrete types; subtype variance is irrelevant to the
   wire format. Add when a use case demands it.

4. **Where-clauses / type-class bounds** (`X[T : SomeContract]`).
   Constraint-bounded templates are useful but require a separate
   constraint-checking pass. Out of scope.

5. **Defaulted type-params** (`X[T = i32]`). Sugar that does not affect
   the materialised wire format.

6. **Field-position instantiation** (`field: Foo[Bar]`). Forbidden by
   locked decision #3. The negative-test matrix #1 makes this a
   compile-time error.

7. **Nested instantiation in alias RHS** (`type Y = Foo[Bar[i32]]`).
   Forbidden by locked decision #3. Negative-test matrix #2.

8. **Per-language reified generics.** No backend emits a generic type;
   every backend emits the monomorphised concrete type. C# does NOT
   produce `Page<int>` in the generated source — it produces
   `IntPage`. Same for Java/Kotlin/TypeScript/Rust generics. This is the
   meaning of decision #4.

9. **Template evolution as a template.** Locked decision #5: only the
   materialised instances participate in `BaboonEvolution`. Evolving a
   template's body affects every alias instantiation downstream;
   evolution diff is computed against each materialised
   `DomainMember.User` independently, exactly as today.

10. **Rust serde-style derive composition for templates.** Out of scope
    because no backend sees the template — decision #4.
