# M19 — BAB-A02: User-defined DTOs as JSON map keys — Plan

## Executive summary

Today, declaring a baboon field of type `map[Wrapper, V]` where `Wrapper` is a user-defined `data` (e.g. a single-field newtype around `uid`/`str`/`i64`) is rejected at codegen time by every JSON backend with `BUG: Unexpected key usertype`, forcing consumers to fall back to `map[str, V]` and lose static typing at the wire boundary. M19 relaxes this restriction by allowing user-defined DTOs (and type aliases / `id` types from M18) to appear as JSON map keys *iff* they recursively ground in a single-primitive field of an already-eligible builtin scalar; the wrapper's existing value-position JSON-object codec is left untouched and a new dedicated map-key codec path is added per backend that delegates to the inner primitive's existing key codec. UEBA backends require no map-key changes (binary keys are positional bytes already handled by the recursive `mkEncoder`/`mkDecoder`), so the work is concentrated in the validator, the typer's `BaboonRuntimeCodec` map-key dispatcher, and the 9 `*JsonCodecGenerator.scala` files plus a cross-language fixture extension.

## 1. Current-state audit

### 1.1 Rejection sites in JSON codec generators

The crash originates in two parallel sites inside every JSON generator: the **encoder** key-dispatcher and the **decoder** key-dispatcher. The Scala reference (excerpt from `ScJsonCodecGenerator.scala:300-327`):

```scala
private def encodeKey(tpe: TypeRef, ref: TextTree[ScValue]): TextTree[ScValue] = {
  tpe.id match {
    case TypeId.Builtins.tsu | TypeId.Builtins.tso => q"$baboonTimeFormats.format($ref)"
    case _: TypeId.Builtin                         => q"$ref.toString"
    case uid: TypeId.User =>
      domain.defs.meta.nodes(uid) match {
        case u: DomainMember.User =>
          u.defn match {
            case _: Typedef.Enum    => /* ...emit enum_JsonCodec... */
            case f: Typedef.Foreign => /* ...follow BaboonRef alias or emit foreign codec... */
            case o => throw new RuntimeException(s"BUG: Unexpected key usertype: $o")  // <-- rejection
          }
        case o => throw new RuntimeException(s"BUG: Type/usertype mismatch: $o")
      }
    case o => throw new RuntimeException(s"BUG: Unexpected key type: $o")
  }
}
```

Equivalent decoder rejection at `ScJsonCodecGenerator.scala:441-460`. Verified the same shape (separate `encodeKey`/`getKeyDecoder` helpers each with their own user-type match and `Unexpected key usertype` throw) exists in:

Note: the TS and Python generators do not throw `Unexpected key usertype`. TS reuses `mkJsonEncoder/Decoder` recursively (which handles user types but emits a JSON object not a string — so a wrapper key today would silently produce non-string JSON keys, broken JSON). Python relies on Pydantic `model_dump`-style serialization. Both still need explicit fix-up for the wrapper case so the emitted JSON key is a primitive string — the bug surface is just different (silent corruption rather than a generator throw). Audit these two more deeply during PR M19.2.

Each file has TWO encoder sites and TWO decoder sites (per generator), one near the top-level `mkEncoder` map case and one near a `getKeyDecoder` inside the field-decoder builder. Both must be updated coherently.

### 1.2 Runtime codec (`BaboonRuntimeCodec.scala`)

`baboon-compiler/.../typer/BaboonRuntimeCodec.scala:548-622` (encode) and `:625-697` (decode) implement runtime JSON↔UEBA cross-format facade map-key handling. Same structure: builtin scalars handled inline; `TypeRef.Scalar(u: TypeId.User)` only accepts `Typedef.Enum` and `Typedef.Foreign` (with `runtimeMapping` fallback); everything else fails with `RuntimeCodecIssue.UnsupportedMapKeyType`. This is the runtime equivalent of the per-backend codegen restriction and **must be relaxed in lockstep** — otherwise `BaboonCodecsFacade.jsonToUebaBytes` / `uebaToJson` will still reject wrapper keys at runtime even after codegen accepts them.

### 1.3 Validator (`BaboonValidator.scala`)

There is **no current validator rule** that rejects a user DTO as a map key. The existing rule, `checkComplexMapKeys` (`BaboonValidator.scala:291-312`), only rejects keys of `TypeRef.Constructor` shape (i.e. nested collections like `map[lst[X], V]`). User-DTO keys reach codegen unfiltered today, which is precisely why `ScJsonCodecGenerator.scala:320` throws — the validator silently lets the bad model through. The companion rule `checkAnyAsMapKey` (`:451-468`) blocks `any` as a key (recursively).

This means PR M19.1 must **add** a new validator check, not relax an existing one — there is no relaxation candidate. The plan calls this `checkUserMapKeysEligibility`.

### 1.4 Type aliases

`Domain.aliases: List[AliasInfo]` (`typer/model/Domain.scala:19`) is populated by `BaboonTranslator.resolveAliasInfo` (`BaboonTranslator.scala:43-55`), which stores `resolvedTarget: TypeRef`. Field-level `TypeRef`s in DTOs are emitted by the typer with aliases already resolved to their target (`ScopeSupport.scala:309` follows aliases through scope resolution), so by the time the validator and codec generators see a field, an alias to `uid` is already a `TypeRef.Scalar(TypeId.Builtins.uid)`. **Spot-verify** during PR M19.1 that `data Foo { value: MyAlias }` where `alias MyAlias = ItemId` (where `ItemId` is itself a wrapper) materialises as the wrapper's `TypeRef.Scalar(<ItemId>)`.

### 1.5 UEBA generators

`ScUEBACodecGenerator.scala:511-525` (decoder) / `:594-600` (encoder): the `Builtins.map` case calls `mkEncoder(c.args.head, ...)` / `mkDecoder(c.args.head)` recursively. The recursive helper already handles user DTO types (it emits the wrapper's standard codec). UEBA does **not** require special key shape — keys are written as raw bytes. **Confirmed**: no UEBA codec generator changes are needed in this milestone.

## 2. Eligibility rule design

### 2.1 Eligible as JSON map key (after M19)

A `TypeRef` `t` is **eligible as a JSON map key** if it is reducible by the procedure `reduceKey(t)` to a `TypeRef.Scalar(builtinScalar)` where the scalar is in the existing primitive-key allowlist (i.e. excluding `f32`/`f64`/`f128` — see §2.3) and the reduction terminates without revisiting an already-seen `TypeId`.

```
reduceKey(TypeRef.Scalar(b: TypeId.BuiltinScalar)) = b                            // base case
reduceKey(TypeRef.Scalar(u: TypeId.User)) =
  match domain.defs.meta.nodes(u).defn:
    Typedef.Enum                                               => Eligible
    Typedef.Foreign with BaboonRef(target)                     => reduceKey(target)
    Typedef.Foreign with runtimeMapping = Some(target)         => reduceKey(target)        // runtime-codec parity
    Typedef.Dto(_, fields, contracts) where contracts.isEmpty
                                       && fields.size == 1
                                       && fields.head.tpe is not opt[_], not collection
                                       && u is in `domain.roots`                => reduceKey(fields.head.tpe)
    Typedef.Id(...) (M18)                                      => same DTO rule applied to id's single-primitive field
    _                                                           => Ineligible
reduceKey(TypeRef.Constructor(_, _))                            => Ineligible
reduceKey(TypeRef.Any)                                          => Ineligible       // already enforced by checkAnyAsMapKey
```

Aliases never appear here because they are pre-resolved by the typer (§1.4).

### 2.2 Ineligible (explicit, document for printer messages)

- Multi-field DTOs (no canonical string repr, even if all fields are primitives).
- DTOs with `contracts` (inheritance changes the on-wire shape).
- DTOs whose single field is `opt[_]` (an option-around-primitive has no stable key — the absent case has no string form).
- DTOs whose single field is a collection (`lst[_]`, `set[_]`, `map[_,_]`).
- ADTs, contracts, services.
- Floats (`f32`/`f64`/`f128`) — same rationale as float-equality semantics for keys; current builtin-key handling already accepts them with caveats. **Open question — see §8.2.**
- Wrapper of `any` (already excluded via `checkAnyAsMapKey`).
- Cyclic wrappers (`data A { v: B }`, `data B { v: A }`) — recursion sees the same `TypeId` twice → reject.

### 2.3 Numeric float caveat

The existing builtin map-key dispatcher *does* accept f32/f64/f128 keys (Scala line 431-433, 482-484; same in other backends). The user spec is silent on tightening this; recommend keeping the current "floats allowed in builtin position, also allowed in wrapper position" symmetry — do not silently break existing fixtures. Flag in §8.

## 3. Codec emission design

### 3.1 JSON: separate map-key codec path per wrapper

**Critical insight:** the wrapper's existing `JsonCodec` (used for value-position fields) emits a JSON OBJECT, e.g. `{"value": "abc"}`. That shape is **wrong** for a JSON map key, which must be a string. The fix is to emit, per wrapper, a parallel "map-key codec" that:

- **Encode** (extract the inner primitive value from the wrapper instance, then call the inner primitive's existing key-encoder):
  ```scala
  // pseudo-emission for wrapper W { value: uid }
  encodeKey(W, w) = encodeKey(uid, w.value)
  ```
- **Decode** (read the JSON map's string key, parse as inner primitive, construct wrapper):
  ```scala
  decodeKey(W, s) = W(decodeKey(uid, s))
  ```

Recursion: if the inner field is itself a wrapper, the same rule applies — no new code needed in the user generator, just recursion of the existing helper.

**Per-backend implementation skeleton** (Scala ref):

In `ScJsonCodecGenerator.encodeKey`, replace the rejection at line 320 with:
```scala
case d: Typedef.Dto if isEligibleKeyWrapper(d) =>
  val inner = d.fields.head
  encodeKey(inner.tpe, q"$ref.${inner.name.name}")   // recurse
case d: Typedef.Dto =>
  throw new RuntimeException(s"BUG: Unexpected key usertype (validator should have rejected): $d")
```

Symmetrically in `getKeyDecoder` (line 456): build a `circeKeyDecoder.instance(s => innerKeyDecoder.decode(s).map(W(_)))`.

### 3.2 Per-backend hook locations

For each backend, the existing **builtin** key-codec emission helpers serve as the precedent (they already handle uid, str, i64, etc. as JSON-string keys); the wrapper extension is the same pattern recursively peeling one wrapper layer. See section 1.1 for the file/line table.

For TS/Python, the change is to add an explicit wrapper unwrap step before the existing primitive-key logic kicks in (so the JSON key is the primitive's string form, not the wrapper's object form).

### 3.3 Runtime codec parity

`BaboonRuntimeCodec.encodeMapKey` (`:548`) and `decodeMapKey` (`:625`): add a `case Typedef.Dto(_, fields, _) if isEligibleKeyWrapper(d) => recurse on fields.head.tpe` arm before the catch-all `F.fail(UnsupportedMapKeyType)`. The runtime codec is part of the JSON↔UEBA cross-format facade and **must accept the same keys the validator allows**.

### 3.4 Wrapper's normal value-position codec — unchanged

The wrapper's standard `JsonCodec` (emitting `{"value": "abc"}`) is left alone. Both shapes coexist:
- value-position usage (`data Foo { id: ItemId }`) → object form (existing behaviour).
- key-position usage (`data Foo { idx: map[ItemId, u32] }`) → primitive-string form (new emission).

This is symmetric to how builtins behave today: a `uid` in value position is an object-of-string `Json.fromString(uuid.toString)`, but as a map key it is just the string itself.

### 3.5 UEBA — no change

UEBA emits keys via the recursive `mkEncoder`/`mkDecoder` already. Since wrapper DTOs already have UEBA codecs, the existing recursive emission Just Works.

## 4. Validator changes

### 4.1 New rule: `checkUserMapKeysEligibility`

Add to `BaboonValidator.checkPostTyper` traversal (alongside `checkComplexMapKeys` at line 228). Pseudocode:

```scala
private def checkUserMapKeysEligibility(domain: Domain, dto: Typedef.Dto, meta: RawNodeMeta)
    : F[NEList[BaboonIssue], Unit] = {
  def isEligibleKey(t: TypeRef, seen: Set[TypeId]): Either[Reason, Unit] = t match {
    case TypeRef.Scalar(b: TypeId.BuiltinScalar) =>
      if (b in floatScalars && !FLOATS_AS_KEYS_ALLOWED) Left(...) else Right(())
    case TypeRef.Scalar(u: TypeId.User) if seen.contains(u) =>
      Left(CyclicWrapper(u))
    case TypeRef.Scalar(u: TypeId.User) =>
      domain.defs.meta.nodes(u) match {
        case U(Enum, _, _, _)    => Right(())
        case U(Foreign(_, _, runtimeMapping = Some(t2)), _, _, _) => isEligibleKey(t2, seen + u)
        case U(Foreign(...), _, _, _)                              => Right(())  // foreign codec drives it
        case U(d: Dto, _, _, _) if d.contracts.isEmpty
                                 && d.fields.size == 1
                                 && !isOptionOrCollection(d.fields.head.tpe) =>
          isEligibleKey(d.fields.head.tpe, seen + u)
        case _ => Left(IneligibleUserType(u))
      }
    case _: TypeRef.Constructor => Left(CollectionAsKey)
    case _: TypeRef.Any         => Left(AnyAsKey)
  }

  val badFields = dto.fields.collect {
    case f if f.tpe.matches(map(k, v)) && isEligibleKey(k, Set.empty).isLeft => f
  }

  F.when(badFields.nonEmpty)(F.fail(...))
}
```

### 4.2 New issue type

`VerificationIssue.IneligibleUserMapKey(owner, badFields, reason, meta)` in `parser/model/issues/TyperIssue.scala`. Printer message must surface the *reason* (multi-field DTO, collection field, ADT, cyclic, etc.) so users get an actionable error instead of a runtime exception.

### 4.3 Bad-fixture matrix for the `m19-bad/*.baboon` cluster

Mirror the `any-bad/*.baboon` pattern from PR 1.4:

- `multi-field-wrapper.baboon`  — `data K { a: u32; b: str }` used in `map[K, i32]`.
- `collection-wrapper.baboon`  — `data K { v: lst[u32] }` used in `map[K, i32]`.
- `option-wrapper.baboon`  — `data K { v: opt[uid] }` used in `map[K, i32]`.
- `adt-key.baboon`  — `adt K = ...` used as map key.
- `cyclic-wrapper.baboon`  — `data A { v: B }` / `data B { v: A }` used as map key.
- `non-root-data-wrapper.baboon` — clarify scope per §8.5.

## 5. Type alias resolution

Aliases are pre-resolved by `BaboonTranslator.resolveAliasInfo` (`BaboonTranslator.scala:43-55`) into `AliasInfo.resolvedTarget: TypeRef`, and field-level `TypeRef`s in DTOs are emitted with aliases already resolved to the target. Therefore:

- `alias ItemId2 = ItemId` followed by `map[ItemId2, V]` reaches the validator as `map[<resolved ItemId>, V]` — the eligibility check applies to the resolved target.
- `alias U64Key = u64` followed by `map[U64Key, V]` reaches the validator as `map[u64, V]` and is already eligible (builtin).

**Verification step in M19.1:** add a positive fixture `wrapper-via-alias.baboon` that exercises `alias K = ItemId; ... map[K, u32]` and verifies it generates the same codec as direct `map[ItemId, u32]`.

## 6. Identifier types interaction (BAB-A01 / M18)

BAB-A01 introduces an `id` keyword that produces a `Typedef.Id` (or augments `Typedef.Dto` with an "is-id" flag — depends on M18 plan). The `id` type's canonical string repr per the M18 task description is `<Name>:<ver>#fields:values:{nested}` — i.e. it can be a *parseable composite* with multiple fields, not just a primitive newtype.

**Recommendation: ship M19 BEFORE M18.**

Rationale:
1. M19 is strictly smaller — relax one validator rule, add per-backend wrapper-key codec, extend runtime codec, add cross-language fixture.
2. M19 does not depend on the `id` keyword.
3. M19's eligibility rule (single-primitive-field DTO) is **strictly narrower** than M18's id repr (which can be multi-field). Making M18 inherit M19's eligibility is a one-line addition: any `Typedef.Id` whose shape is a single-primitive field automatically passes the existing M19 rule.
4. If M18 ships first, the `id` types not all of them would be map-key eligible (multi-field ids must be rejected), so the eligibility rule in M19 still has to discriminate by shape — same code regardless of order.

## 8. Open questions for the user

1. **Wrapper-of-wrapper recursion depth:** `data StrongUserId { v: UserId }; data UserId { v: uid }` ⇒ recurse twice. Recommendation: yes, no depth limit beyond cycle detection. **Confirm.**

2. **Floats as wrapper-key inner type** (`data Score { v: f64 }` as map key): current builtin path already accepts `f64` keys. Recommendation: keep parity — accept float wrappers iff floats are accepted as builtin keys. **Confirm or tighten globally.**

3. **Both shapes coexist:** wrapper's standard JSON-object codec (value-position) + new map-key codec (key-position). Per §3.4. **Confirm acceptable.**

4. **Non-root `data` as map-key wrapper:** the user spec said "root-data wrappers". Test whether non-root `data` can transitively be a map key today (likely yes via `lst[Foo]`). Recommendation: scope eligibility to **root-data only**. **Confirm.**

5. **`opt`-around-primitive in wrapper field:** `data MaybeId { v: opt[uid] }` — recommendation: reject. **Confirm.**

6. **`id` types (M18 interaction):** ship M19 first, M18 inherits map-key eligibility for single-primitive id shapes. **Confirm sequencing.**

7. **Foreign types with `runtimeMapping`:** today these recurse into the runtime-mapped target. Recommendation: same rule applies in M19 — a wrapper around a Foreign whose `runtimeMapping` resolves to a primitive is eligible. **Confirm.**

8. **Validator vs codegen failure mode:** today the model is rejected at codegen with `BUG: Unexpected key usertype`. After M19, an *ineligible* wrapper is rejected at validator time with a clean error. The codegen `BUG: Unexpected …` throw is kept as a defensive assertion. **Confirm.**

## Note on scope discipline

This plan deliberately does NOT touch:
- UEBA codec generators (no change needed; verified §3.5).
- The wrapper's value-position JSON-object codec (left intact; coexists with new map-key codec).
- ADT / contract / service map-key handling (all remain rejected).
- The `checkAnyAsMapKey` / `checkComplexMapKeys` validator rules (orthogonal).
- BAB-A01 `id` keyword (M18 — separate plan).
