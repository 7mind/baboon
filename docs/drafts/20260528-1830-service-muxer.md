Status: draft, awaiting confirmation
Scope: TypeScript first, then propagate to Scala/C#/Python/Rust/Kotlin/Java/Dart/Swift

## Problem

The current generated `dispatchJson` / `dispatchUeba` (per-domain) takes an `impls: { svc1, svc2, â€¦ }` record and requires the caller to provide *every* service of that domain at construction time. There is no convenient way to combine services Ă  la carte â€” especially across domains.

## Design (confirmed)

- **Cross-domain**: a single `JsonMuxer` / `UebaMuxer` can hold services from any model.
- **Interface-based routing**: each generated *service wrapper class* implements `IBaboonJsonService` / `IBaboonUebaService`. The muxer dispatches on `serviceName()`.
- **TS first**, then mechanically applied to the other 8 backends.

## Concrete shape (TS)

### Runtime additions (`BaboonSharedRuntime.ts`)

```ts
export interface IBaboonJsonService {
    readonly serviceName: string;
    invoke(method: BaboonMethodId, data: string, ctx: BaboonCodecContext): Promise<string>;
}

export interface IBaboonUebaService {
    readonly serviceName: string;
    invoke(method: BaboonMethodId, data: Uint8Array, ctx: BaboonCodecContext): Promise<Uint8Array>;
}

export class JsonMuxer {
    private readonly table = new Map<string, IBaboonJsonService>();
    constructor(...services: IBaboonJsonService[]) {
        for (const s of services) this.register(s);
    }
    register(s: IBaboonJsonService): void {
        if (this.table.has(s.serviceName)) {
            throw new BaboonWiringException({ tag: 'DuplicateService', serviceName: s.serviceName });
        }
        this.table.set(s.serviceName, s);
    }
    invoke(method: BaboonMethodId, data: string, ctx: BaboonCodecContext): Promise<string> {
        const s = this.table.get(method.serviceName);
        if (s === undefined) {
            throw new BaboonWiringException({ tag: 'NoMatchingService', method });
        }
        return s.invoke(method, data, ctx);
    }
}

export class UebaMuxer { /* analogous */ }
```

### Per-service generated emission (alongside existing `invokeJson_X` / `invokeUeba_X`)

```ts
export class VirtualEnvironmentAdminServiceJsonService implements IBaboonJsonService {
    constructor(private readonly impl: VirtualEnvironmentAdminService) {}
    readonly serviceName = "VirtualEnvironmentAdminService";
    invoke(method: BaboonMethodId, data: string, ctx: BaboonCodecContext): Promise<string> {
        return invokeJson_VirtualEnvironmentAdminService(method, data, this.impl, ctx);
    }
}

export class VirtualEnvironmentAdminServiceUebaService implements IBaboonUebaService {
    constructor(private readonly impl: VirtualEnvironmentAdminService) {}
    readonly serviceName = "VirtualEnvironmentAdminService";
    invoke(method: BaboonMethodId, data: Uint8Array, ctx: BaboonCodecContext): Promise<Uint8Array> {
        return invokeUeba_VirtualEnvironmentAdminService(method, data, this.impl, ctx);
    }
}
```

### User-facing usage

```ts
const json = new JsonMuxer(
    new VirtualEnvironmentAdminServiceJsonService(adminImpl),
    new ReportServiceJsonService(reportImpl),
);
await json.invoke({ serviceName: "VirtualEnvironmentAdminService", methodName: "shutdown" }, payload, ctx);
```

## Coexistence

- Existing `dispatchJson` / `dispatchUeba` (per-domain) are kept. They are useful when the caller wants the record-of-impls ergonomics.
- The muxer is the new composable variant.

## Service-result modes

For `noErrors=false` modes (where the dispatcher takes an additional `rt: IBaboonServiceRt`), the muxer wrapper also accepts an `rt` in its constructor and threads it through. The wrapper class definition adapts to the same axes.

## Service-context mode

The existing `ctx` parameter encodes both `BaboonCodecContext` and an optional service context (Abstract/Concrete). The muxer keeps this contract â€” the wrapper forwards `ctx` straight through.
