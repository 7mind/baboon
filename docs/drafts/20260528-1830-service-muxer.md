# Service Muxer — design draft

Status: draft, awaiting confirmation
Scope: TypeScript first, then propagate to Scala/C#/Python/Rust/Kotlin/Java/Dart/Swift

## Problem

The current generated `dispatchJson` / `dispatchUeba` (per-domain) takes an `impls: { svc1, svc2, … }` record and requires the caller to provide *every* service of that domain at construction time. There is no convenient way to combine services à la carte — especially across domains.

## Design (confirmed)

- **Cross-domain**: a single `JsonMuxer` / `UebaMuxer` can hold services from any model.
- **Interface-based routing**: each generated *service wrapper class* implements `IBaboonJsonService` / `IBaboonUebaService`. The muxer dispatches on `serviceName()`.
- **TS first**, then mechanically applied to the other 8 backends.

## Concrete shape (TS)

### Runtime additions (`BaboonSharedRuntime.ts`)

```ts
export interface IBaboonJsonService {
    readonly serviceName: string;
    invoke(method: BaboonMethodId, data: string, ctx: BaboonCodecContext): Promise<string>;
}

export interface IBaboonUebaService {
    readonly serviceName: string;
    invoke(method: BaboonMethodId, data: Uint8Array, ctx: BaboonCodecContext): Promise<Uint8Array>;
}

export class JsonMuxer {
    private readonly table = new Map<string, IBaboonJsonService>();
    constructor(...services: IBaboonJsonService[]) {
        for (const s of services) this.register(s);
    }
    register(s: IBaboonJsonService): void {
        if (this.table.has(s.serviceName)) {
            throw new BaboonWiringException({ tag: 'DuplicateService', serviceName: s.serviceName });
        }
        this.table.set(s.serviceName, s);
    }
    invoke(method: BaboonMethodId, data: string, ctx: BaboonCodecContext): Promise<string> {
        const s = this.table.get(method.serviceName);
        if (s === undefined) {
            throw new BaboonWiringException({ tag: 'NoMatchingService', method });
        }
        return s.invoke(method, data, ctx);
    }
}

export class UebaMuxer { /* analogous */ }
```

### Per-service generated emission (alongside existing `invokeJson_X` / `invokeUeba_X`)

```ts
export class VirtualEnvironmentAdminServiceJsonService implements IBaboonJsonService {
    constructor(private readonly impl: VirtualEnvironmentAdminService) {}
    readonly serviceName = "VirtualEnvironmentAdminService";
    invoke(method: BaboonMethodId, data: string, ctx: BaboonCodecContext): Promise<string> {
        return invokeJson_VirtualEnvironmentAdminService(method, data, this.impl, ctx);
    }
}

export class VirtualEnvironmentAdminServiceUebaService implements IBaboonUebaService {
    constructor(private readonly impl: VirtualEnvironmentAdminService) {}
    readonly serviceName = "VirtualEnvironmentAdminService";
    invoke(method: BaboonMethodId, data: Uint8Array, ctx: BaboonCodecContext): Promise<Uint8Array> {
        return invokeUeba_VirtualEnvironmentAdminService(method, data, this.impl, ctx);
    }
}
```

### User-facing usage

```ts
const json = new JsonMuxer(
    new VirtualEnvironmentAdminServiceJsonService(adminImpl),
    new ReportServiceJsonService(reportImpl),
);
await json.invoke({ serviceName: "VirtualEnvironmentAdminService", methodName: "shutdown" }, payload, ctx);
```

## Coexistence

- Existing `dispatchJson` / `dispatchUeba` (per-domain) are kept. They are useful when the caller wants the record-of-impls ergonomics.
- The muxer is the new composable variant.

## Service-result modes

For `noErrors=false` modes (where the dispatcher takes an additional `rt: IBaboonServiceRt`), the muxer wrapper also accepts an `rt` in its constructor and threads it through. The wrapper class definition adapts to the same axes.

## Service-context mode

The existing `ctx` parameter encodes both `BaboonCodecContext` and an optional service context (Abstract/Concrete). The muxer keeps this contract — the wrapper forwards `ctx` straight through.

## Outstanding questions (please confirm)

1. **Wrapper class name**: `<ServiceName>JsonService` and `<ServiceName>UebaService` (proposal above) — or some other suffix to avoid collisions with hand-written client classes? Current generation already emits `<ServiceName>Client`; I picked `JsonService` / `UebaService` to mirror it. ✅ / suggest alternative
2. **Error type**: muxer throws `BaboonWiringException({ tag: 'NoMatchingService', method })` on unknown service. The existing `dispatch*` throws `BaboonWiringException({ tag: 'NoMatchingMethod', method })`. Should the muxer use the same `NoMatchingMethod` tag (no new tag) — or is a dedicated `NoMatchingService` tag preferred? ✅ / `NoMatchingMethod` instead
3. **Duplicate-registration**: muxer's `register` throws on duplicate `serviceName`. Alternative: silently overwrite. Strict-by-default matches the rest of baboon's "fail fast" tone. ✅ / overwrite
4. **Async / sync coupling**: TS muxer's `invoke` is `Promise<…>`. When `--ts-async-services` is false, services return sync values today — the wrapper would `Promise.resolve()`-wrap them, OR the muxer becomes sync-only. Easiest: pin the muxer to async (matches `BaboonCodecsFacade` style and is what the user has set anyway). ✅ / sync mode too

## Test coverage (2026-05-29)

Cross-language muxer test coverage now exists for all 9 codec backends:

- **TS / C# / Scala / Rust / Python** — muxer cases (route-to-I1, route-to-I2,
  NoMatchingService, DuplicateService, both JSON + UEBA) appended to the
  existing `*-wiring-either` overlays (errors mode, `BaboonEither`/native
  `Either`/`Result` container). Validated by the existing
  `test-{cs,sc,rs,py,ts}-wiring-either` actions.
- **Kotlin / Java / Dart / Swift** — new dedicated wiring harnesses:
  overlay dirs `test/{kt,jv,dt,sw}-stub-wiring-overlay/`, gen+test actions
  `test-gen-{kt,jv,dt,sw}-wiring` / `test-{kt,jv,dt,sw}-wiring` in
  `.mdl/defs/tests.md`, registered in the `test` aggregator. These run in
  **no-errors** service-result mode (`--service-result-no-errors=true`,
  matching the service-acceptance harness) and run only the wiring test class
  so the stubs' other handwritten tests need not be re-validated.

Harness gotchas captured while building these:
- Each new-language gen action must replicate that language's
  `test-gen-regular-adt` flags exactly (else the stub's handwritten tests fail
  to compile). In particular Java needs `--jv-write-evolution-dict=true`
  (gates per-domain `BaboonMetadata` emission used by `BaboonExtTest` etc.).
- Dart: the generated runtime files must be `mv`'d into
  `packages/baboon_runtime/lib/` and the checked-in `test/runtime/` restored
  post-codegen (mirrors regular-adt).
- Swift: the codegen-emitted `CrossLanguageFixturePath.swift` must be fanned
  out into each `Tests/BaboonTests/<Module>/` subdir; the muxer test lives in
  the hand-written `Tests/RuntimeTests/` target, which needed `TestpkgPkg0`
  added to its dependency list in `test/sw-stub/Package.swift`.
- Swift's `IBaboonJsonService` is an associated-type protocol, so muxer
  registration goes through the `AnyJsonService<R>` / `AnyUebaService<R>`
  type-erasers. Rust's muxer `invoke` returns `Result<R, BaboonWiringError>`
  (outer = routing, inner = the service's own `R`).
