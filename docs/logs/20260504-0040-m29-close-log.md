## PR-29.10 critical discoveries

### Discovery 1 ŌĆö Latent Python ADT JSON codec defect surfaced by m29

**Symptom**: `M29OkHolder_JsonCodec.encode()` produced `{"Ok":{}}` instead of `{"Ok":{"value":42}}` for ADT variants with fields. Pre-M29 CI was green because no existing fixture had registered an ADT-with-field-bearing-variants in `test/conv-test-py/compat_main.py` ŌĆö the path was never exercised cross-language. m29-ok is the first fixture to register `IntStrEnvelope` (materialised from `adt Envelope[T, E] { data Ok { value: T } data Err { error: E } }`).

**Root cause**: pydantic v2's inherited `@model_serializer(mode='wrap')` on the ADT base class (`IntStrEnvelope`) dispatched `serializer(self)` using the parent's field schema (zero fields), not the runtime subclass's schema (`Ok` has `value: int`). The serializer was structurally correct for the parent but silently wrong for variants.

**Fix**: `PyJsonCodecGenerator.scala` extended with `fieldHasAdt` / `dtoHasAdtField` helpers; ADT scalar branch in `mkJsonAnyEncoder` / `mkJsonAnyDecoder` now delegates to the variant's own `_JsonCodec.instance().encode/decode` rather than calling pydantic's `model_dump_json()`. The variant codec produces the correct `{"Variant": {fields}}` shape. Symmetric on encode and decode.

**Verification**: `mdl :test-acceptance` 200/200 (was 145/200 pre-fix). The 55 newly-passing rows are exactly the previously-broken Python ADT cases (Python-as-source for IntStrEnvelope to all 8 other sinks + all sources to Python-as-sink + roundtrip combinations).

### Discovery 2 ŌĆö Baboon services constrained to ONE input arg per shorthand method

**Symptom**: D02 brief specified `def put (K, V): unit` for the new `Crud[K, V]` service template. Round-2 executor used `def put (V): K` instead.

**Investigation**: `DefService.scala` `shorthandSig` rule (L42-48) is `"(" ~ shorthandRef("in") ~ ")" ~ ":" ~ shorthandRef("out")` ŌĆö exactly one input. `BaboonTranslator.scala` `convertMethod` (L430, L433) enforces `inargs.size > 1` ŌåÆ `ServiceMultipleInputs` error and accepts `inargs.head`, not a list. Maximum one input per method is a hard architectural constraint of the shorthand path. Multi-arg requires the struct-arg path (`def name { in { a: A; b: B } out { ŌĆ” } }`). The `unit` type does not exist in Baboon.

**Disposition**: D02 brief was based on incorrect assumptions. The current shape `def put (V): K` is the only valid 2-type-param shorthand exercising both type params. Documented in `[PR-29.10-D08]` resolution; constraint added to "future work must respect" notes.

### Discovery 3 ŌĆö `:test-service-acceptance` does not exercise the templated service end-to-end

**Symptom**: Round-2 fix added `Crud[K, V]` to `baboon-compiler/src/test/resources/baboon/m29-ok/m29.baboon` (the typer-test fixture). `:test-service-acceptance` 81/81 PASS ŌĆö but that's the petstore.baboon baseline; petstore has no template.

**Disposition**: D02's typer-side path is verified by `TemplateInstantiatorTest` 12/12. End-to-end cross-language service-acceptance for templated services is a separate sub-task: would require extending `test/services/petstore.baboon` with a templated service + updating 9 per-language service harnesses. Deferred to `[PR-29.10-D07]` explicit follow-up.
