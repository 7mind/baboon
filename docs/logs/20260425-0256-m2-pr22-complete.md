## Key surprises / lessons from this session

1. **PR 1.4's `any-bad/` fixtures broke `mdl` codegen actions** (D11). Discovered during PR 2.1 verification, fixed in PR 2.0 (single mechanical relocation). Lesson for future negative-fixture PRs: keep them outside any directory that doubles as a codegen root.
2. **Reserved meta-kind bytes silently accepted** (PR-04-D01). Initial PR 2.1 `AnyMeta.require` only checked bitâ†”Option alignment, not that `kind` was one of the six locked values. Future language ports must reject `0x04` and `0x05` explicitly â€” silent acceptance is a forward-compat hazard. Fix added a `VALID_KINDS` set and a fourth `require`.
3. **`Set[Byte]` Scala gotcha**: `Set(0x00, 0x01, ...)` is `Set[Int]`, not `Set[Byte]`. Must explicitly `.toByte` each element. Used in PR-04-D01's fix.
4. **`readJson` error-channel discipline** (PR-04-D02): aligning `readJson` to return `Either[BaboonCodecException, AnyMeta]` (not throw) saves PR 2.3 churn. Binary path still throws â€” UEBA is trust-the-wire.
5. **Spec Â§172 helper factoring matters at scale** (PR-05-D02): inlining the codec template at every site yielded 737-line generated DTOs. Per-codec helpers cut to 282 lines (61% reduction). Future language milestones should follow the same pattern.
6. **Decoder must honor `meta-length`** (PR-05-D01): blindly reading meta then immediately reading blob silently corrupts against future meta extensions. Counting-stream wrapper + `skipBytes` is the right shape; future language ports need the equivalent.
7. **Cross-format encoding is architectural** (deferred to PR 2.3): codec objects in Scala are static `object` singletons with no facade reference; `BaboonCodecContext` is a `useIndices` flag, not a registry. Threading a facade through requires either (a) extending `BaboonCodecContext`, (b) a global registered facade singleton, (c) a method-signature change. Not resolvable as a PR-2.2-internal decision.
8. **`AnyOpaqueUeba` array-equality defect** (PR-05-D08): JVM case-class default `equals` uses reference identity on `Array[Byte]`. Latent in PR 2.1; surfaced in PR 2.2 by the first generated `_tests.scala` round-trip. Override via `java.util.Arrays.equals/hashCode`. Future language ports with similar binary-blob ADTs must address content equality up-front.
9. **Discipline deviation noted**: PR-05-D08 fix was made directly by the orchestrator (10-line edit), not via fix subagent, after two consecutive 529 API overloads blocked subagent dispatch. Acceptable degraded-mode workaround; recorded in `defects.md` D08 and here. Should not become a pattern.

## What's next â€” blocker for PR 2.3

**PR 2.3** (Scala JSON codec emission) is symmetric to PR 2.2: emit JSON codec encode/decode for `TypeRef.Any` per the spec's `{$ak, $ad?, $av?, $at?, $c}` envelope. Scope of what changes is well-defined.

**However**: PR 2.3 must also resolve the cross-format encoding question PR 2.2 deferred. Both PR 2.2 (which currently `throw`s on `AnyOpaqueJson` input) and PR 2.3 (which will `throw` on `AnyOpaqueUeba` input unless cross-convert lands) need a way to reach the facade from inside generated codec code. The codec objects are static singletons; reaching a domain-specific facade for `(domain, version, typeid)` lookup is the hard part.

### Three options for the user to pick from

**(a) Extend `BaboonCodecContext`** to carry an optional `BaboonCodecsFacade` reference, plumbed through every codec method that may need cross-convert. Cleanest type-wise but touches many code-gen sites; affects every language port.

**(b) Global registered facade singleton** in the runtime, with a `baboon.runtime.shared.GlobalFacade.get()` accessor. Application registers one at boot. Codecs reference the singleton when cross-convert is needed. Simple to land in Scala first; potentially controversial (singleton-as-DI smell). Other languages map straightforwardly.

**(c) Cross-convert as a separate code path on `AnyOpaque` itself** â€” make `AnyOpaque` a method-bearing class with `def asUeba(facade)` / `def asJson(facade)`. Codec generators emit a call that asks the user-supplied `AnyOpaque` to convert itself, threading the facade through the encode-call's outer scope. Cleaner separation but pushes user-API complexity.

**(d) Ban cross-format encoding for variant D fields**, require the user to pre-resolve. PR 2.3 still emits `RuntimeException` for cross-format like PR 2.2 does. Documented limitation. Smallest scope but reduces ergonomic value of `AnyOpaque` (defeats part of the spec's motivation).

The plan Â§171 named the helpers (`uebaToJson`/`jsonToUeba`) but didn't say where they live or how generated code reaches them.

**Question for user**: which option (or another)?
