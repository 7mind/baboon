## Why the loop returned

Per review-loop skill rule I6: "fundamental plan flaw that needs user judgement" → mark `[!]`, escalate to session end.

The plan's PR 1.2 scope is fine in intent (typed AST + typer + minimal translator shims) but wrong in surface estimate (64 cascade sites vs. implied ~9). The user should decide between:

**(a) One huge mechanical PR 1.2 — recommended.** 64 sites × ~3 lines each ≈ 200 lines of trivial `case _: TypeRef.Any => throw new RuntimeException("BUG: `any` unsupported for <language> until M<N>")` additions. Keeps every commit consistent. Violates CLAUDE.md §5 in surface but not in depth.

**(b) Split into two commits with a broken build in between.** PR 1.2a = typed AST + typer + compiler tests (cascade sites broken). PR 1.2b = 64-file placeholder cascade. Cleanliness argument, but deliberate broken-build commits are risky.

**(c) Represent `any` without a new sealed `TypeRef` case.** Reuse `TypeRef.Constructor(TypeId.Builtins.any, args)` with synthetic encoding of variant + underlying. No cascade; compromises typed-AST honesty.

**(d) Move variant + underlying off `TypeRef` onto `Field`.** `tpe: TypeRef.Scalar(any)`; `anyMeta: Option[AnyMeta]` sibling on the field. No cascade; splits field semantics across two places.
