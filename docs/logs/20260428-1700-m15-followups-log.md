## Process notes

- **Investigate before plan**: dispatching the planner with explicit "investigate the dart failure FIRST" was load-bearing — turned what looked like an enum-related issue into a u64-overflow issue, which is a completely different fix surface. A planning subagent that started from "fix the enum spec" and dragged the dart fix in would have wasted significant time.
- **Plan errors caught by executors**: the original Dart u64 plan suggested `$ref.toUnsigned(64).toString()`, but Dart's `int.toUnsigned(64)` returns the same negative `int` for negative values. The executor caught this via live Dart 3.11.4 and switched to `BigInt.from($ref).toUnsigned(64).toString()`. Reinforces the value of executors who actually run the code rather than trust the plan.
- **`.capitalize` semantics are subtle**: `bar_pub.capitalize == "Bar_pub"`, NOT `"BarPub"`. Easy to misread. PR-35-D03 unit test pins this against future "fixes" by contributors who think `wireName` should do real Pascal-case conversion.
- **Cross-language coverage matters**: PR-35 succeeded only because the executor ran the FULL matrix (`:test-gen-compat-*` for all 9 languages). Per-language regression alone would have missed wire-format divergences. PR-35-D06 (deferred) is the next step — extend any-showcase compat fixture to include non-Pascal-case enums for byte-identity verification.
- **Wire-format breaks**: PR-35 introduced two real breaks not in any reporter's complaint — TS/Dart/Swift change from raw to Pascal, and C#/JVM-runtime drop case-insensitive decode. Both documented in the new spec doc's Migration Implications section. Existing deployments accepting non-Pascal-case enum strings will need to update.
